## 역량 신호 (관찰)
1. **감사 프로세스가 성숙 궤도에 진입.** 3라운드 연속(7/14·15·17)에서 🔴 블로커가 (3 라이브 뚫림)→(드리프트 시정)→**0**으로 수렴. 직전 하드닝이 라이브에 생존 = "고쳤다"와 "배포됐다"의 간극이 닫히는 중.
2. **라이브 실측(§3-D)이 감사의 authoritative 축으로 정착.** 이번 5개 감사 전부 소스가 아닌 라이브(anon 프로브·프로덕션 SHA 대조·게이트 실호출·advisor 소멸)로 판정. 7/15에 적발된 소스-배포 드리프트가 이번엔 처음부터 0(웰코 master==store==prod SHA 일치).
3. **Fable 격리 병렬 감사 패턴 효과 실증.** 5개 독립 레포를 격리 컨텍스트 5명이 동시 감사 → Alpha 컨텍스트 오염 0, 파일충돌 0, 회장 지시 "페이블 활용" 충족. 위임 트리거(T1/T2/T3/T5) 전부 발화한 정당한 위임.

## 개선 제안 (propose-only)
- **P1 — ASD 스캐너에 ReDoS 룰 신설.** SF-1(스캐너 자기 정규식 ReDoS)은 스캐너가 스캐너를 못 잡은 사각지대. `scanner/rules/`에 "무제한 수량자(`*`/`+`가 부정문자클래스 `[^...]`에 붙은) 패턴" 탐지 룰 추가 후보. 카탈로그 T-07에 이미 사례 등록됨.
- **P2 — 배포 프로브 자동화.** T-23은 정적분석 불가 → 감사 때마다 수동 curl. "프로덕션 SHA vs HEAD 대조 + 라이브 보안헤더 스모크"를 스크립트/CI화하면 드리프트 상시 감지(수동 의존 제거 = 회장 자동화 원칙과 정합).
- **P3 — Supabase 버킷 정책 표준 템플릿.** F-CIN-06(public 버킷 SELECT 폴더 무스코프)은 신데렐라 고유가 아닌 BaaS 반복 함정. G2 표준 "public 버킷 SELECT = 본인폴더 스코프" 템플릿을 신규 프로젝트 게이트(g2-new-project)에 편입 후보.
