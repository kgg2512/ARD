# ARD 아키텍처 — 증거로 오르는 루프 사다리 (Evidence-Gated Loop Ladder)

> **개정:** 2026-07-02 (Alpha, 회장 지시) · **성격:** ARD 구조 개편의 정본. README는 이 문서를 요약·링크한다.
> **한 줄:** ARD를 "3개 축(파악·검증·개선)"에서 **"검증이 게이트인 하나의 루프 사다리"**로 재정의한다. 지식은 Harness가 증거로 통과시키기 전엔 자동화하지 않는다.

---

## 0. 왜 다시 그리는가 (지난 세션의 정직한 진단)

지난 세션 회장 보고 요지 [확인 — 이 세션 파일 재확인]:
- ARD의 최대 성과 = "스킬·MCP 인프라가 사실상 허수(다수 0회 호출)"임을 **증거로 폭로**한 것.
- 플래그십 `ard-loop`은 **미설치·미검증 → 현재 자율 생산량 0.** "작동하는 시스템"이 아니라 **"배선된 코드."**
- G2의 진짜 단단한 자산은 딱 셋 — **execution_log + 검증 하드게이트 + CAE.** 나머지는 서류상.
- ARD는 지금 "개선 엔진"이 아니라 **"진단·감사 도구."**

그리고 회장의 **메타 불확실성**: *"스킬표 vs Opus 직접, 어느 게 더 좋고 효율적인지 아직 모른다."* → 그래서 지금 결정하지 않는다. **이 개편의 목적은 결정이 아니라, 그 불확실성을 스스로 해소하는 구조를 만드는 것.**

## 1. 세상의 "루프" 담론 vs G2 (대조)

2026 SNS·Anthropic이 합의한 프레임: **"이제 프롬프트가 아니라 루프다"(Karpathy).** 루프 = *멈춤 조건이 충족될 때까지 반복하는 에이전트.* Claude Code팀이 정의한 **4단계**는 "사람이 무엇을 넘기느냐"로 나뉜다.

원문 인사이트 전문: [`ard-loop/docs/insights/2026-07-02_ai-loops-and-stacks.md`](./ard-loop/docs/insights/2026-07-02_ai-loops-and-stacks.md).

### 핵심 발견 — G2는 이 4단계를 **이미 흩어진 채로 전부** 굴리고 있다

| 루프 단계 | 사람이 넘기는 것 | 원문 도구 | **G2의 기존 대응물** [확인] | 상태 | ARD에서의 역할 |
|---|---|---|---|---|---|
| **1. 턴 기반** | 검수 | 맞춤 검증 SKILL.md | **Alpha 직접 + 검증 하드게이트**(매 작업) | ✅ **작동** — G2 유일 단단 자산 | ARD가 실제 가치를 낸 지점(진단·감사) |
| **2. 목표 기반** | 종료조건 | `/goal` (evaluator) | **Harness A/B**(정지조건까지 실험) + 작업지시서 §4 evaluator 루프(PASS까지) | ⚠️ 수동이나 작동 | **Harness = ARD의 Loop 2 엔진(증거 게이트)** |
| **3. 시간 기반** | 트리거 | `/loop`·`/schedule` | git-sync SessionStart 훅 · 큐레이터 7일 · influencer 일1회 | 🟡 일부 작동 / ❌ ard-loop 미설치 | `ard-loop` HARVEST = 스케줄 수확(설치 대기) |
| **4. 능동형** | 프롬프트 전체 | 위 전부 + 동적 워크플로 | **CAE 자동발동**(Stop 훅·이벤트·사람0) | ✅ **CAE 작동** / ❌ ard-loop OrgDev 미도달 | ard-loop 완전자율 = 증거 통과 후에만 승격 |

> **이 표가 개편의 출발점.** G2는 루프 인프라가 없어서 못 도는 게 아니다. **4단계를 전부 갖고 있다.** 특히 **CAE는 G2가 이미 라이브로 굴리는 유일한 Loop 4(능동형)** — 이벤트(10건 누적/세션종료)로, 사람 개입 0으로, 자동 발동된다.

### CAE가 증명하는 것 (개편의 열쇠) [추정 — 해석]

CAE는 Loop 4가 **작동한다**는 산 증거다. 단, 왜 작동하나? **과제가 좁고(rule 준수 여부만) + 검증 가능**하기 때문. 반면 `ard-loop`의 자율 "수확→이해→적용"은 **넓고 + 신뢰만으로 검증 불가**. → **좁고 검증 가능하면 Loop 4가 돈다. 넓고 신뢰 기반이면 멈춘다.** 이게 ard-loop이 배선 상태에 갇힌 진짜 이유다.

## 2. "하네스"란 무엇이며 G2의 하네스는 어디 있나

SNS의 JARVIS/OpenClaw류가 파는 것 = **하네스**(LLM을 감싸는 실행 골격 = 루프 + 컨텍스트 + 툴 + 검증). *같은 모델이라도 하네스 설계가 품질을 가른다.*

**G2의 하네스는 이미 세계급이다 — JARVIS 음성 장난감보다 낫다:**

| 하네스 부품 | 루프의 어느 조각 | G2 자산 | 상태 |
|---|---|---|---|
| 프롬프트/규칙 | (루프 진입) | CLAUDE.md 헌법 | ✅ |
| 컨텍스트/메모리 | gather context | execution_log · wiki · mem0 | ✅ 단단 |
| 툴 | take action | 스킬 600+ · MCP 19 | ⚠️ **다수 0회 호출(허수)** |
| **검증** | **verify the work** | **검증 하드게이트 + g2-qa-tester + CAE** | ✅ **단단 (핵심 자산)** |

> **결론: G2에 없는 건 하네스가 아니다. "루프를 증거로 올리는 규율"이다.** 회장이 부러워할 게 JARVIS의 음성이 아니라, 정작 우리가 가진 검증 하네스를 **루프로 안 돌리고 있다**는 사실. 툴 층(스킬)이 허수인 이유도 여기 — 툴을 라우팅할 **증거 게이트가 루프에 안 걸려 있어서.**

## 3. 핵심 개편 — 증거로 오르는 루프 사다리

### 원칙 (한 줄): **"검증(Loop 2)이 통과시킨 것만 자동화(Loop 3/4)로 승격한다."**

지난 세션의 실패를 한 문장으로 재정의하면: **ard-loop은 Loop 1/2에서 가치를 증명하기 전에 Loop 3/4(자율 수확·적용)로 건너뛰었다.** 이건 인사이트 문서가 인용한 Karpathy의 실패패턴 그 자체다:
- **Kitchen Sink** — 수도꼭지(스킬 하나 검증) 고치랬더니 자율 수확 공장 전체를 지음.
- **Optimistic Path** — "수확한 지식은 당연히 좋을 것"이라는 예외 무시. Harness가 이걸 반증했다(스킬 강제 +0.35 = 동전던지기, 42%는 오히려 나쁨).

### 사다리 (증거가 각 단을 열어야 다음 단으로)

```
 Loop 1 턴기반         Loop 2 목표기반           Loop 3 시간기반         Loop 4 능동형
 (검증 하드게이트)  ──▶ (Harness 증거 게이트)  ──▶ (스케줄 수확·실행)  ──▶ (완전 자율)
 매 작업 강제           ★여기가 ARD의 심장★         증거 통과분만            좁고+검증가능만
 ✅ 이미 작동           ⚠️ 수동·확장 필요           🟡 HARVEST만 안전         ✅ CAE만 도달
                        │                                                    ▲
                        └──── 증거 없으면 승격 차단 (게이트) ─────────────────┘
```

- **Loop 2 = 게이트.** 어떤 지식/스킬/조직변경도 Harness식 증거(정지조건까지 측정)를 통과하기 전엔 Loop 3/4로 못 올라간다.
- **HARVEST(수확)만은 예외적으로 Loop 3 즉시 허용** — 수확은 큐를 채울 뿐 적용이 아니다(공급망 게이트가 이미 자동설치를 막음). 위험 0.
- **APPLY(적용)·PROMOTE(의무화)는 게이트 뒤.** 방법론형 지식의 "의무화"는 Harness가 양(+)의 효과를 낼 때만.

## 4. ARD 재정의 — "개선 엔진"이 아니라 "증거 엔진"

기존 서사(과장): *"CAR이 세상 지식을 자율로 흡수·적용·자가개선하는 루프."* → 이게 배선 상태로 멈춘 원인(넓고 신뢰 기반).

**개편 서사(정직):** ARD = **G2가 무엇에 투자할지를 증거로 판정하는 엔진.** 3개 축은 유지하되, **검증을 축들 사이의 게이트로 승격**한다.

```
 ① 파악·학습            →   ② 평가·검증 (게이트)      →   ③ 개선 (승격)
 Loop 3 스케줄 수확          Loop 2 Harness 증거          Loop 1→3→4 증거 통과분만 자동화
 "싸게 많이 줍는다"          "이게 진짜 효과 있나"          "증명된 것만 손에 쥔다"
 ─ HARVEST는 자유            ─ 통과 못하면 여기서 폐기        ─ 조직변경도 propose→증거→반영
```

이게 회장의 메타 불확실성을 **구조로 해소**한다: "스킬 vs 직접"을 *지금* 결정할 필요 없다. **Loop 2(Harness)를 계속 돌려 증거를 쌓고, 헌법은 증거가 임계를 넘기 전까지 동결.** ARD가 곧 그 증거 엔진이고, 출력이 루프를 승격시킨다.

## 5. 구체적 개편 3가지 (지식 · 조직역량 · 구성)

### 개편 A — 지식(Knowledge): "수집"이 아니라 "증거 태깅"

- 문제: `docs/insights/`·스킬 레지스트리가 **효과 미측정 상태로 쌓임**(수집=가치 착각).
- 개편: 모든 인사이트·적용 항목에 **3-태그** 부착 → `수확됨 / 적용됨 / 측정된 효과(Δ or 미측정)`. 측정 안 된 지식은 "후보"일 뿐 자산 아님.
- 강제 장치(이미 보유): `tool_usage_audit.py`(0회 호출 탐지) + 큐레이터(7일 미사용 아카이브). ARD의 파악축은 **줍기만 하지 말고 call-rate를 측정**한다. → 지식은 **얇게 유지**(회장 PC경량·헌법비대화금지와 정합).

### 개편 B — 조직역량(Capability): CAR = "개선자"가 아니라 "제안자+측정자"

- 문제: CAR 서사가 "적용·자가개선"으로 과장 → propose-only 경계와 긴장, 그리고 방법론 자동적용은 Harness가 위험하다고 증명.
- 개편: **CAR의 APPLY를 두 종류로 쪼갠다**(CAR.md 반영):
  - **도구 설치형(신뢰출처+1K+):** 지금처럼 자동 허용 — 설치는 되돌리기 쉽고 공급망 게이트가 있다.
  - **방법론 의무화형("이 스킬을 매 작업 강제"):** **Harness 증거 없이는 제안(propose)까지만.** systematic-debugging(+3.0, 호출≈0)처럼 **증거가 이미 있는 것만 라우팅 강화**, ponytail·ui-ux-pro-max(무·음수)는 opt-in 강등 — **단 반영은 회장 승인 후.**
- 순효과: CAR이 "매 리뷰마다 헌법을 흔드는" 자기참조 폭주를 **증거 게이트가 구조로 차단.** CAR은 제안하고, Harness가 채점하고, 회장/Alpha가 처분.

### 개편 C — 구성(Composition): 폴더는 그대로, **루프 레벨 라벨을 입힌다**

- 원칙: 폴더 물리 재배치 = 순수 처른(churn) → **안 한다**(ponytail). 대신 각 하위 프로젝트에 **루프 레벨 + 상태 배지**를 단다(README 반영):

| 하위 | 루프 레벨 | 상태 | 다음 한 걸음 |
|---|---|---|---|
| 검증 하드게이트(G2 헌법) | **L1** | ✅ 작동 | 유지 — ARD의 기반 |
| **Harness** | **L2 게이트** | ⚠️ 수동·1차완료 | 3차 실험(회장 승인) = 게이트 정밀화 |
| ard-loop HARVEST | **L3** | ❌ 미설치 | 설치 = 안전(큐만 채움) |
| ard-loop APPLY/OrgDev | **L4** | ❌ 미도달 | 게이트 뒤 — Harness 증거 후 |
| CAE | **L4** | ✅ 작동 | 유지 — Loop 4 작동 증명 |
| git-sync | **L3** | ✅ 작동 | 유지 |
| Claude-App-Factory / insights | 지식(L3 입력) | 🟢 축적 | 증거 태깅(개편 A) |

### 개편 D — Creation 종착: Hermes 접합 (2026-07-04, 회장 지시)

- **문제(실측):** ARD 수확물은 **불활성 큐**(자막·레포)로 끝나 `status=pending`만 쌓임(45건, 인지단계 사실상 0회 실행). Hermes 무인루프와 비교하니 결정적 차이 = **Hermes 산출물은 태생이 실행체(스킬)**, ARD는 지식→스킬 **증류 단계가 없음.**
- **개편: ARD를 별도 루프가 아니라 G2 스킬 루프(=Hermes curator 이식본)의 "외부 입력 팔"로 접합.**
  1. 수확(HARVEST) → 큐 적재 (현행 유지)
  2. **실제 작업이 당김(Pull):** Alpha가 실작업 중 "더 나은 기법?" 시 큐를 조회 (push 아님)
  3. **검증 통과 시 증류:** 그 기법이 실작업의 검증 게이트를 통과하면 → `.claude/skills/ard-<이름>/SKILL.md`로 박음. frontmatter: `g2_origin: agent-created` + `g2_created:<날짜>` + `g2_source: ard-harvest` + **`g2_verified: true`**(실작업 통과 = 측정된 효과 → 큐레이터 보호대상, 희귀해도 아카이브 안 됨).
  4. **큐레이터 관할:** 이후는 기존 `curator.py`가 사용량으로 관리(코드 변경 0 — 이미 `g2_origin: agent-created` 관리·`g2_verified:true` 보호).
- **튜닝(Hermes와 다르게):** Hermes 큐레이터 게이트=사용빈도. ARD발 스킬은 희귀하나 결정적일 수 있어 **게이트=사용빈도 아니라 "실작업 검증 통과 사건"**(`g2_verified:true`로 표현) → 조기 아카이브 방지.
- **순효과:** 상주 0·제2두뇌 0·헌법 위반 0으로 Hermes 무인루프를 네이티브 재현. ARD = 세상의 지식을 G2 스킬 위장(curator)에 넣는 앞단. Creation 강제력은 헌법 완료 게이트(스킬화 판정)가 담당 → [[reference-hermes-creation-gate]].

## 6. 결정은 미루되, 구조는 준비 (회장 결정 최소화)

회장은 지금 아무것도 결정하지 않아도 된다. 하지만 이 개편은 **다음 결정을 작고 안전하게** 쪼갠다:

- **기존 블로커:** "ard-loop 설치할까?"(설치=schtasks 4개+SessionStart 훅=상주, 회장 PC경량 원칙과 긴장) → **크고 무서운 하나의 결정.**
- **개편 후:** 블로커를 **둘로 분리** —
  1. **L3 HARVEST만 설치**(큐만 채움, 적용 0, 되돌리기 쉬움) = **작고 안전한 실험.** 회장이 원할 때.
  2. **L4 자동적용/OrgDev 상주** = **Harness가 "수확 지식이 양의 효과"를 증명한 뒤에만.** 지금은 propose-only 동결.
- 즉 **"자율 공장을 켤까"라는 큰 결정 대신, "수확 큐를 켜볼까"라는 작은 결정**으로 낮췄다. 그것도 지금 강요 안 함.
